Component Architecture

Réutilisation​

Vous pouvez diviser un composant en plusieurs composants plus petits pour avoir une conception plus lisible et maintenable et / ou pour obtenir une réutilisation.

Mais comment savez-vous quel devrait être son propre composant?

Voici quelques questions à vous poser lors de la détermination du moment de créer un autre composant:

Est-il possible que votre morceau de code soit réutilisé?

  • Si oui, la construction d'un nouveau composant semble être une excellente idée.
  • même si la réutilisation est dans un seul composant.

Votre code est-il assez complexe?

  • Si oui, c'est peut-être une bonne idée de se diviser dans des composants séparés afin de rendre votre code plus lisible et maintenable.

Conception de logiciels

  • En général, utilisez simplement les mêmes techniques pour décider si vous devez créer une nouvelle fonction ou un nouvel objet.
  • Une de ces techniques est le principe de responsabilité unique, c'est-à-dire qu'un composant ne devrait idéalement faire qu'une seule chose.S'il finit par croître, il doit être décomposé en sous-composants plus petits.

Après avoir créé plus de composants, plus de questions se posent telles que:

  • Comment les composants doivent-ils interagir?(Communication des composants)
  • Y a-t-il des modèles de conception que je devrais suivre lors de la création de composants?(Levage de l'état vers le haut, composition vs héritage)
  • Quels types de composants y a-t-il?(Conteneur vs présentation)
  • Comment rendre mes composants réutilisables?

Cette section explore chacune de ces questions.

Communication des composants​

Comment les composants doivent-ils interagir?

Modèles de communication communs​

Les composants communiquent généralement de cette manière:

  • parent de l'enfant
  • Enfant à parent

La communication du parent à l'enfant passe une propriété de données dans un composant.Plus précisément, passer certaines données (pourrait être une chaîne (primitive), un objet, un tableau) dans un composant enfant.

function App() {
  return <Parent />;
}

function Parent() {
  const [words, setWords] = React.useState('');

  const handleClick = () => {
    setWords('Did you do your homework?');
  };

  return (
    <div>
      <h1>Parent</h1>
      <button onClick={handleClick}>Ask</button>
      <Child hears={words} />
    </div>
  );
}

function Child(props) {
  return (
    <div>
      <h2>Child</h2>
      <p>{props.hears}</p>
    </div>
  );
}

ReactDOM.createRoot(document.getElementById('root')).render(<App />);

L'enfant à parent La communication passe une fonction en tant que propriété dans un composant.La fonction est ensuite invoquée chez l'enfant mais exécutée dans le contexte du parent.

function App() {
  return <Parent />;
}

function Parent() {
  const handleRequest = (request) => {
    if (request.includes('car')) {
      alert('No');
    }
  };

  return (
    <div>
      <h1>Parent</h1>
      <Child onRequest={handleRequest} />
    </div>
  );
}
function Child(props) {
  const handleClick = () => {
    props.onRequest('Can I have the car?');
  };

  return (
    <div>
      <h2>Child</h2>
      <button onClick={handleClick}>Ask for the car</button>
    </div>
  );
}

ReactDOM.createRoot(document.getElementById('root')).render(<App />);

Modèles de communication supplémentaires​

Maintenant que vous comprenez comment cette communication fonctionne dans la pratique, nous pouvons explorer ces modèles de communication supplémentaires qui sont essentiellement des variations sur "Parent to Child" et "Enfant to Parent"

  • Enfant à enfant (frères et sœurs)
  • Grand-parent à petit-enfant
  • petit-enfant à parent

Essentiellement, La communication ne saute pas les générations, donc si cela va se produire, vous devez communiquer manuellement vers le haut ou vers le bas entre chaque génération.

La communication des composants ne fonctionne pas comme les événements JavaScript ... il n'y a pas d'événements.

Modèles de conception

Levage State Up​

Souvent, plusieurs composants doivent refléter les mêmes données changeantes.Nous vous recommandons de soulever l'état partagé jusqu'à leur ancêtre commun le plus proche.

Voici un exemple.

Exemple de composant de fonction​

function Button({ onClickFunction }) {
  return <button onClick={onClickFunction}>+1</button>;
}

const Result = ({ value }) => {
  return <div>Result: {value}</div>;
};

function App() {
  const [counter, setCounter] = React.useState(0);

  const incrementCounter = () => {
    setCounter((previousCounter) => previousCounter + 1);
  };

  return (
    <div>
      <Button onClickFunction={incrementCounter} />
      <Result value={counter} />
    </div>
  );
}

ReactDOM.createRoot(document.getElementById('root')).render(<App />);

Exemple de composant de classe​

class Button extends React.Component {
  render() {
    return <button onClick={this.props.onClickFunction}>+1</button>;
  }
}

const Result = (props) => {
  return <div>Result: {props.value}</div>;
};

class App extends React.Component {
  state = {
    counter: 0,
  };

  incrementCounter = () => {
    this.setState((prevState) => ({
      counter: prevState.counter + 1,
    }));
  };

  render() {
    return (
      <div>
        <Button onClickFunction={this.incrementCounter} />
        <Result value={this.state.counter} />
      </div>
    );
  }
}

ReactDOM.createRoot(document.getElementById('root')).render(<App />);

La documentation React le résume le mieux:

Il devrait y avoir une seule «source de vérité» pour toutes les données qui changent dans une application React.Habituellement, l'état est d'abord ajouté au composant qui en a besoin pour le rendu.Ensuite, si d'autres composants en ont également besoin, vous pouvez le soulever à leur ancêtre commun le plus proche.Au lieu d'essayer de synchroniser l'état entre les différents composants, vous devez compter sur le flux de données descendant.

Le levage de l'état consiste à écrire plus de code «passe-partout» que les approches de liaison bidirectionnelles, mais comme avantage, il faut moins de travail pour trouver et isoler les bogues.Étant donné que tout état «vit» dans un composant et que ce composant seul peut le changer, la surface des insectes est considérablement réduite.De plus, vous pouvez implémenter n'importe quelle logique personnalisée pour rejeter ou transformer l'entrée utilisateur.

Si quelque chose peut être dérivé des accessoires ou de l'état, il ne devrait probablement pas être dans l'état.

Composants de conteneur et de présentation​

Composants de conteneur (intelligent)​

  • sont préoccupés par comment les choses fonctionnent
  • Fournir les données et le comportement aux composants de présentation ou d'autres conteneurs
  • charge et modifie les données via les appels vers une API
  • peut contenir à la fois des composants de présentation et de conteneur à l'intérieur, mais n'a généralement pas de balisage DOM à l'exception de quelques divs emballages, et jamais de styles
  • CONNAISSEZ également sous le nom de composants de conteneurs ou de composants du contrôleur

Composants de présentation​

  • sont préoccupés par à quoi ressemblent les choses
  • Recevez des données et des rappels exclusivement via des accessoires
  • Ne spécifiez pas comment les données sont chargées ou modifiées
  • CONNAISSEZ également sous le nom de composants purs ou de composants stupides

Référence​

  • Presentational and Container Components by Dan Abramov

  • Not a Rule but Something to Notice

Jusqu'à récemment, lorsque React introduit «Hooks» semble que presque tous les composants ont finalement besoin de l'état afin que les gens aient eu tendance à créer des composants de classe par défaut.En réalité, ce n'est pas l'un ou l'autre ... souvent des composants sont un mélange des deux.Sachez simplement que plus ils deviennent stades, plus ils sont difficiles à tester et à réutiliser, mais la réutilisation a souvent un coût de complexité.

Souvent, mes composants de présentation ne sont pas des composants de présentation purs, ils ont un état d'État particulièrement local.

Composition vs héritage​

React recommande d'utiliser la composition au lieu de l'héritage pour réutiliser le code entre les composants.

En général, les composants peuvent être imbriqués à l'intérieur d'autres composants ou vivre à côté d'autres composants comme dans HTML où un <div> peut avoir un <p> à l'intérieur et le<p> peut avoir un <A> et un<Mg>.HTML fonctionne sur la composition, donc les composants réagissent fonctionnent sur la composition car ils sont essentiellement des balises HTML que vous avez inventées.

See the section Composition vs Inheritance in the React documentation for more information.

Nous utilisons React dans des milliers de composants, et nous n'avons trouvé aucun cas d'utilisation où nous recommandons de créer des hiérarchies sur l'héritage des composants.- Facebook

Penser dans React​

Voici quelques étapes que vous pourriez trouver utiles lorsque vous apprenez pour penser dans React

  1. Cassez l'interface utilisateur en une hiérarchie de composants

  2. Créez une version statique dans React

    • Aucun état ou accessoire

  3. Identifiez la représentation minimale (mais complète) de l'état de l'interface utilisateur

  4. Identifiez où votre état devrait vivre

    • Pour chaque élément d'état de votre application:

      • Identifiez chaque composant qui rend quelque chose en fonction de cet état.
      • Trouvez un composant propriétaire commun (un seul composant au-dessus de tous les composants qui ont besoin de l'état dans la hiérarchie).
      • Le propriétaire commun ou un autre composant plus élevé dans la hiérarchie doivent être propriétaires de l'État.
      • Si vous ne trouvez pas de composant où il est logique de posséder l'état, créez un nouveau composant simplement pour maintenir l'état et l'ajouter quelque part dans la hiérarchie au-dessus du composant propriétaire commun.

  5. Ajouter un flux de données inverse

    • Le rendu de l'écran implique initialement des accessoires et de l'État coulant dans la hiérarchie
    • Le flux de données inverse fait référence aux composants au plus profond de la hiérarchie répondant aux actions de l'utilisateur (cliquant sur un bouton, en survol, en tapant), puis en mettant à jour l'état dans le ou les composants de conteneurs supérieurs

See the section Thinking in React in the documentation for more information.
